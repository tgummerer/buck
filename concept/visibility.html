Visibility

Visibility determines whether a build rule can include a build target in its list of deps. In a large project, you may want to prevent developers from "reaching across" the project and pulling in additional code. Reducing the visibility of a build rule can help prevent that type of behavior.

Build rules are visible to the other targets inside the same build file by default.

To make a build rule visible to other targets, add a visibility argument to the build rule and provide a list of build target patterns.

There is also a special value, 'PUBLIC', which makes a build rule visible to all other rules.

Examples

A common library like Guava should be able to be included by any build rule:
prebuilt_jar(
  name = 'guava',
  binary_jar = 'guava-14.0.1.jar',
  visibility = [
    'PUBLIC',
  ],
)
It is common to restrict the visibility of Android resources to the Java code that uses it:
android_resource(
  name = 'ui_res',
  res = 'res',
  package = 'com.example',
  visibility = [
    '//java/com/example/ui:ui',
  ],
)
Or it may be simpler to make it visible to the entire directory in case additional build rules are added to java/com/example/ui/BUCK:
android_resource(
  name = 'ui_res',
  res = 'res',
  package = 'com.example',
  visibility = [
    '//java/com/example/ui:',
  ],
)
Finally, it is common to limit code for testing to be visible only to tests. If you define all of your Java unit tests in a folder named javatests/ in the root of your project, then you could define the following rule to ensure that only allow build rules under javatests/ can depend on JUnit:
prebuilt_jar(
  name = 'junit',
  binary_jar = 'junit-4.11.jar',
  visibility = [
    '//javatests/...',
  ],
)
